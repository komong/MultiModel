# 知识库模板字段说明

> 本文档对 `kb-template.md` 中每个字段和结构的用途、填写规范进行详细说明，
> 帮助团队成员正确维护知识库词条。

---

## 一、YAML Frontmatter 字段说明

这部分在文件最顶部，被 `---` 包裹，是给程序（脚本/向量数据库）读的元数据。
**人工填写，脚本可批量校验和更新。**

---

### `id`

```yaml
id: kb-0042
```

- **用途**：词条的全局唯一标识符，供脚本索引、词条互相引用使用
- **规范**：统一格式 `kb-` + 4位数字，从 `kb-0001` 开始递增
- **注意**：一旦创建不要修改，其他词条的 `related` 字段会引用这个 id
- **脚本用途**：嵌入时作为向量数据库的主键，防止重复入库

---

### `title`

```yaml
title: Node.js 18 安装与配置
```

- **用途**：词条的正式名称，会被 agent 检索时优先匹配
- **规范**：简洁、包含关键词，避免"关于xxx的说明"这类冗余表达
- **建议**：带上版本号（如有），方便区分同类词条

---

### `category`

```yaml
category: 软件配置
```

- **用途**：词条分类，用于 metadata 过滤，agent 可按类别缩小检索范围
- **可选值**（固定选一个，保持一致）：

| 值 | 适用场景 |
|---|---|
| `系统环境` | 操作系统设置、环境变量、路径配置 |
| `硬件配置` | 开发机、服务器的硬件规格记录 |
| `软件配置` | 编程语言运行时、数据库、中间件安装配置 |
| `开发工具` | IDE、编辑器、CLI 工具、插件配置 |
| `代码维护` | 代码规范、重构记录、模块说明、接口文档 |
| `部署运维` | CI/CD、Docker、服务器部署、监控 |
| `故障排查` | 已解决的 Bug、报错处理、性能问题 |

---

### `tags`

```yaml
tags: [Node.js, nvm, 环境配置, macOS]
```

- **用途**：多维度关键词，是向量检索的重要 metadata 过滤字段
- **规范**：
  - 3 ~ 6 个为宜，不要太多（稀释权重）也不要太少
  - 包含：技术名称、平台、版本关键词、操作类型
  - 统一用中文或英文，团队内保持一致（推荐技术名词用英文）
- **脚本用途**：批量统计标签分布，发现知识库覆盖盲区

---

### `platform`

```yaml
platform: macOS
```

- **用途**：标记词条适用的操作系统，避免 agent 把 Linux 命令推给 Windows 用户
- **可选值**：`macOS` / `Windows` / `Linux` / `跨平台`
- **注意**：如果一个配置在三个平台都不同，建议拆成三个词条，不要混写

---

### `arch`

```yaml
arch: arm64
```

- **用途**：标记 CPU 架构，在 Apple Silicon（M 系列）和 x86 共存的团队里非常关键
- **可选值**：`x86_64` / `arm64` / `通用`
- **典型场景**：Docker 镜像、某些编译工具在不同架构下安装方式不同

---

### `status`

```yaml
status: active
```

- **用途**：控制词条是否参与检索，是知识库"保鲜"的核心字段
- **可选值**：

| 值 | 含义 |
|---|---|
| `active` | 当前有效，正常参与检索 |
| `deprecated` | 已过时（如旧版本），脚本可自动过滤，不推给 agent |
| `draft` | 草稿，内容未完成，暂不参与检索 |

- **维护建议**：版本升级后，旧版词条改为 `deprecated` 而不是删除，保留历史记录

---

### `version`

```yaml
version: "18.20.0"
```

- **用途**：记录词条描述的软件/工具版本号
- **规范**：用字符串格式（加引号），避免 YAML 把 `1.0` 解析成数字
- **适用场景**：软件配置、开发工具类词条必填；硬件、规范类可留空 `""`

---

### `created_at` / `updated_at`

```yaml
created_at: 2026-01-10
updated_at: 2026-05-21
```

- **用途**：
  - `created_at`：创建日期，只填一次，不要修改
  - `updated_at`：每次修改正文后更新
- **规范**：`YYYY-MM-DD` 格式
- **脚本用途**：可自动扫描超过 N 天未更新的 `active` 词条，提示人工复查

---

### `author`

```yaml
author: 张三
```

- **用途**：记录词条创建/主要维护人，出现问题时知道找谁确认
- **规范**：填真实姓名或工号，不要填昵称（流动性强时难以追溯）

---

### `related`

```yaml
related: [kb-0038, kb-0051]
```

- **用途**：关联词条列表，agent 在回答时可主动拉取相关上下文
- **规范**：填 `id`，不要填标题（标题会变，id 不变）
- **脚本用途**：可生成知识库的关联关系图，发现孤立词条

---

### `source`

```yaml
source: "https://nodejs.org/en/download"
```

- **用途**：记录内容的原始来源（官方文档、内部 Wiki、会议纪要等）
- **规范**：外部来源填 URL，内部来源填文件路径或文档名
- **可留空**：原创内容不需要填

---

## 二、正文 Markdown 结构说明

正文是给人和 agent 读的，结构决定分块质量。

---

### 顶部引用块（`>`）

```markdown
> 一句话说明这个词条解决什么问题。
```

- **用途**：词条摘要，agent 检索到词条时优先读这一句判断是否相关
- **规范**：必填，控制在 1 ~ 2 句，不超过 50 字
- **类比**：相当于搜索结果里的 description

---

### `## 环境信息`

```markdown
- **操作系统**：macOS 14.5 Sonoma
- **硬件架构**：Apple M3 Pro
- **依赖版本**：nvm 0.39.7
```

- **用途**：让 agent 判断词条是否适用于当前用户环境
- **规范**：尽量精确到小版本号；跨平台词条可填"见各平台章节"

---

### 分类正文结构

模板提供了 5 种结构对应不同 category，**选一个用，删掉其他的**。

| category | 推荐结构 |
|---|---|
| 系统环境 / 软件配置 | 安装 → 配置 → 验证 |
| 硬件配置 | 规格参数表 → 注意事项 |
| 开发工具 | 用途 → 安装 → 核心配置 → 常用命令 |
| 代码维护 | 背景 → 涉及文件 → 步骤 → 代码示例 |
| 故障排查 | 现象 → 原因 → 解决方案 → 预防措施 |

**为什么按标题分块很重要**：向量数据库切 chunk 时，
`## 安装` 和 `## 配置` 会被切成两个独立 chunk，
用户问"怎么安装"只会召回安装部分，不会夹带无关内容。

---

### 代码块语言标注

````markdown
```bash
brew install node
```

```python
import frontmatter
```

```json
{ "key": "value" }
```
````

- **必须标注语言**，不要用裸的 ` ``` `
- **原因**：agent 读到语言标注后知道这是可执行命令还是配置文件，
  回答时表达更准确；同时渲染工具也依赖这个做高亮

---

### `## 常见问题`

```markdown
**Q：安装后 node 命令找不到？**
A：需要重启终端或执行 `source ~/.zshrc` 使环境变量生效。
```

- **用途**：Q&A 格式和用户提问方式最接近，召回率最高
- **建议**：每个词条写 1 ~ 3 条真实遇到过的问题，不要凑数

---

### `## 相关词条`

```markdown
- [[kb-0038]] nvm 多版本管理
- [[kb-0051]] npm 私有仓库配置
```

- **用途**：双链格式，兼容 Obsidian 等工具直接跳转
- **规范**：先写 id，再写标题，title 变了还能靠 id 找到

---

### `## 变更记录`

```markdown
| 日期 | 变更内容 | 作者 |
|---|---|---|
| 2026-05-21 | 创建 | 张三 |
| 2026-05-28 | 补充 arm64 安装步骤 | 李四 |
```

- **用途**：追溯词条变更历史，不依赖 Git blame 也能看清楚谁改了什么
- **规范**：每次修改正文都追加一行，同步更新 frontmatter 的 `updated_at`

## 三、维护规范建议

### 命名规范

```
kb-0001-node18-install-macos.md
kb-0002-docker-arm64-config.md
```

文件名 = id + 简短描述，方便目录浏览时不打开文件就能判断内容。

### 目录结构建议

```
knowledge_base/
├── active/          # status: active 的词条
├── deprecated/      # status: deprecated 的词条（归档不删）
├── draft/           # status: draft 的草稿
└── _meta/
    ├── kb-template.md       # 模板文件
    └── kb-meta-001.md       # 本说明文档
```

### 脚本可以做的事（后期）

- 批量校验所有词条的必填字段是否完整
- 扫描 `updated_at` 超过 180 天的 `active` 词条，生成待复查清单
- 将 `active` 词条批量解析 → embedding → 入向量数据库
- 自动把 `deprecated` 词条从检索索引中移除
- 统计标签分布，找出覆盖不足的技术领域
